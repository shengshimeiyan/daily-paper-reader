# 1 Introduction 36 

Quantitative systems pharmacology models translate mechanistic understanding of disease biology 37 

into predictions of drug response 1. QSP models can comprise tens to hundreds of ordinary differen-38 

tial equations, and calibration requires data from diverse sources including targeted experimental 39 

studies and clinical endpoints 2,3 . Because virtual population generation and virtual clinical trials 40 

depend on parameter distributions rather than point estimates 4,5 , calibration must quantify un-41 

certainty. As QSP increasingly integrates with machine learning 6,7,8,9,10 , the bottleneck of manual 42 

parameter extraction becomes more acute. 43 

Extracting calibration data from literature requires judgment about applicability, uncertainty, 44 

and potential biases, with full provenance for reproducibility. A structured representation of such 45 

data captures what was measured, under what conditions, with what uncertainty, and how that 46 

measurement relates to model parameters or observables. Some studies isolate specific mechanisms 47 

and directly constrain individual parameters, while clinical and in vivo endpoints constrain the 48 

integrated model through observables computed over the full system state. 49 

Manual curation remains the standard approach but is labor-intensive, with inconsistent docu-50 

mentation across individuals and projects. When personnel change, the reasoning behind parameter 51

Traditional NLP pipelines for automated extraction require task-specific training data 11,12,13 and 54

Large language models offer a more flexible approach. LLMs can identify relevant litera-56

systematic detection: hallucinated values can be caught by requiring verbatim source text, fabri-60 

cated citations through DOI resolution, and unit and code errors by parsing units and executing the 61 

code with mock inputs. This suggests a workflow where LLMs handle extraction while automated 62 

validators catch characteristic errors, with Pydantic 22 , a Python data validation library, enabling 63 

a self-correcting retry loop. 64 

We present MAPLE (Model-Aware Parameterization from Literature Evidence), a framework 65 

for LLM-assisted QSP model calibration. Its core components are: (1) model-aware literature 66 

search using mechanistic context to guide LLM web search; (2) validated YAML schemas for two 67 

calibration regimes (the SubmodelTarget schema for isolated experiments constraining individual 68 

parameters, and the CalibrationTarget schema for clinical/in vivo endpoints constraining the full 69 

model), separating data extraction from model specification while preserving traceability; and (3) 70 

targeted validators for hallucination detection (value-in-snippet matching), citation verification 71 

(DOI resolution), and code correctness. The version of MAPLE used here (v0.1.0) additionally 72 

generates a joint Julia inference script from the validated targets; this inference layer has since 73 

moved to a separate package, but the analysis reported below uses the integrated v0.1.0 pipeline. 74 

The schemas, validators, and reference database do not require an LLM: a modeler can populate 75 

them manually and the validators run against the result the same way. We evaluate MAPLE on 76 

SubmodelTargets and CalibrationTargets for a pancreatic ductal adenocarcinoma (PDAC) QSP 77 

model and report metrics on validation performance and extraction reliability. The LLM produces 78 

usable forward-model choices and observable code when given enough context, but cannot tell when 79 

the context is incomplete, and it can miss inputs or lose track of constraints across long records. 80

# 2 Methods 85 

2.1 Model-Aware Literature Search 86 

Figure 1 summarizes the full MAPLE pipeline. The user specifies which model parameters or 87 

observables need calibration, and the system builds a prompt that guides an LLM with web search 88 

capability to find relevant experimental literature. The framework supports multiple LLM providers 89 

(OpenAI, Anthropic, Google) through a common interface. In this study, we used GPT-5.1 for batch 90 

extraction and Claude Opus 4.6 for interactive extraction. 91 

The prompt injects mechanistic context from the model structure: for each target, the system 92 

retrieves its name, units, and description. For parameters, it also retrieves the reactions where 93 

the parameter appears with rate laws, the species involved, and the broader reaction network. For 94 

observables, it retrieves the species and compartments that define the observable. This context 95 

guides the LLM toward experiments that constrain the targets. Source exclusion prevents reuse 96 

of papers across extractions to promote literature diversity. The prompt also includes the com-97 

plete schema structure and guidance on common pitfalls, reducing the retry loop by preventing 98 

predictable errors. 99 

2.2 Schema Design 100 

The framework defines two complementary schemas for two calibration regimes. The SubmodelTar-101 

get schema captures isolated experiments (in vitro assays, single-mechanism studies) that constrain 102 

individual model parameters through simplified forward models. The CalibrationTarget schema 103 

captures clinical and in vivo endpoints that constrain the full model through observables computed 104 

over the complete species state. Both schemas share a common design philosophy but differ in 105 

how they connect literature data to model parameters and outputs. For clarity, the manuscript 106 

refers to these two schemas by name ( SubmodelTarget and CalibrationTarget ) rather than using 107 

“calibration target” as a general term. 108

2.2.1 Shared Design Principles 109 

Both schemas enforce a data-first architecture: extraction of what the paper reports is separated 110 

from specification of how to use it for calibration. Every extracted numeric value must appear ver-111 

batim in a quoted text snippet from the source, creating an auditable trail and enabling automated 112 

hallucination detection. 113 

2.2.2 SubmodelTarget: Parameter Priors from Isolated Experiments 114 

The SubmodelTarget schema structures each record into two layers: an inputs layer that captures 115 

data exactly as reported in the literature (value, units, uncertainty, sample size, source reference, 116 

and quoted snippet), and a calibration layer that specifies how that data constrains model pa-117 

rameters through a simplified forward model. The submodel derives from the full QSP model 118 

by isolating relevant species. Shared parameter names establish the connection: a proliferation 119 

rate named k apsc prolif in a submodel refers to the same parameter in the full model, allow-120 

ing posteriors to serve directly as priors for full-model calibration. Each parameter includes a 121 

prior distribution with documented rationale, and an error model specifies the likelihood function 122 

connecting predictions to data. Supplementary Section 6 provides detailed field descriptions with 123 

YAML examples. 124 

Forward model type system. The schema provides 15 built-in forward model types, described 125 

in Supplementary Table 1. Each structured model declares its fields as typed roles rather than raw 126 

values, allowing the same template to serve different estimation scenarios depending on which field 127 

is treated as the unknown. 128

2.2.3 CalibrationTarget: Clinical and In Vivo Endpoints 167 

Many calibration constraints come from clinical or in vivo studies where the measurement reflects 168 

integrated model behavior. The CalibrationTarget schema represents these full-model endpoints 169 

through three components. The observable specifies how to compute the measurement from model 170 

species as a Python function that receives the solved state and returns a quantity with units; named 171 

constants carry provenance traced to a curated reference database or literature source. The empir-172 

ical data block derives summary statistics (median and 95% CI) from literature inputs via Monte 173 

Carlo simulation. For intervention studies, a scenario block captures treatment context (agents, 174 

doses, schedules). Complete examples and schema definitions are provided in Supplementary Sec-175 

tions 7 and 5. 176 

2.2.4 Source Relevance Assessment 177 

Experimental data rarely matches the target model perfectly. The source relevance block docu-178 

ments the translation from source context to model context across several dimensions: indication 179 

match, species translation, evidence type, and tumor microenvironment compatibility. Each di-180 

mension requires justification when the source deviates from the target model. An estimated 181 

translation uncertainty fold field quantifies the combined expected uncertainty from all trans-182 

lation factors, which propagates to prior width during inference (e.g., cross-species extraction from 183 

a proxy indication might warrant 10-fold uncertainty). Supplementary Section 13 provides detailed 184 

field descriptions. 185

2.2.5 Schema Implementation 186 

Both schemas are implemented using Pydantic 22 with typed fields and constraints validated auto-187 

matically during YAML parsing. SubmodelTarget selects among its 15 forward model types with 188 

a single type field that determines which remaining fields are required and validated (a Pydantic 189 

discriminated union); CalibrationTarget instead has a single fixed structure, with the same source, 190 

relevance, and context fields on every target. Supplementary Section 5 provides the complete 191 

schema definitions and Supplementary Section 12 provides implementation details. 192 

2.3 Validation Framework 193 

Both schemas share a common validation core with schema-specific extensions. Each validator runs 194 

independently, and a target must pass all validators before proceeding to downstream use. 195 

2.3.1 Reference Validation 196 

LLMs sometimes generate references to entities that do not exist elsewhere in the document. Ref-197 

erence validators check that internal references ( uses inputs , source ref , model parameters, and 198 

observable state variables) all resolve to defined entities. 199 

2.3.2 Content Validation 200 

Content validators address fabrication of values, citations, and other factual claims. DOI valida-201 

tion queries CrossRef for every DOI; title validation further checks that the extracted paper title 202 

fuzzy-matches the CrossRef metadata (75% similarity threshold), catching cases where a valid DOI 203 

is paired with fabricated bibliographic details. Value-in-snippet validation compares each extracted 204 

numeric value against its associated source text, providing the primary anti-hallucination defense: 205 

when an LLM hallucinates a number, it rarely fabricates a consistent snippet, and the mismatch 206 

provides a reliable detection signal. A separate external validator (Section 2.3.3) further verifies 207 

that snippets appear in the cited papers. Unit validation parses all unit strings using Pint 23 to 208 

verify dimensional validity. Code validation executes observation and observable code with mock 209 

data to verify correct behavior, catching logic errors and signature mismatches before inference. 210 

Source relevance validators enforce minimum uncertainty thresholds for cross-species (2-fold), cross-211

2.3.3 External Validation 224 

An additional external validator verifies that value snippets actually appear in the cited source 225 

papers by fetching paper text from Europe PMC and Unpaywall using the DOI, then using fuzzy 226 

matching (80% similarity threshold) to locate each snippet. This catches a subtle failure mode where 227 

the LLM fabricates plausible-looking snippets that contain the correct value but do not actually ap-228 

pear in the cited paper. Because this requires network access to external APIs, it runs as a separate 229 

manual step. It is provided as an on-demand script ( scripts/validate snippets in source.py )230 

in the analysis repository, built on the MAPLE full-text fetch and fuzzy-matching validators, and 231 

was applied selectively during curation rather than as part of the automated per-target valida-232 

tion gate, since external full-text availability is uneven across sources; the value-in-snippet check 233 

described above is the systematically applied internal anti-hallucination defense. 234

2.4 Inference Code Generation 243 

A validated SubmodelTarget already specifies its forward model, priors, and likelihood, so the 244 

same record that passes validation can be turned into inference code without further hand-coding. 245 

MAPLE v0.1.0 (the version used here) includes a translator that emits a Julia script for Turing.jl 25 ;246 

later versions of MAPLE keep the schema and validators and provide the inference translator in 247 

a separate package. The translator joins targets that share parameter names, so each shared 248 

parameter is declared once and constrained by every source that constrains it. Supplementary 249 

Section 4 describes the generated code structure, forward model handling, and inference diagnostics. 250 

The two schemas connect through a natural calibration pipeline: SubmodelTarget posteriors 251 

from isolated experiments become informative priors for full-model calibration against Calibra-252 

tionTarget endpoints. Because SubmodelTarget parameter names match the full model, the poste-253 

rior distributions transfer directly without post-hoc mapping. 254 

2.5 Application: PDAC QSP Model 255 

The PDAC QSP model used in this study is adapted from a spatial QSP model of hepatocellular 256 

carcinoma 26 that first incorporated fibroblast dynamics into the tumor microenvironment. We 257 

evaluate MAPLE on this model using both schemas: 37 curated SubmodelTargets for individ-258 

ual parameter calibration from isolated experiments (spanning tumor growth, stromal dynamics, 259 

immune recruitment, and cytokine signaling), and 45 CalibrationTargets for full-model endpoints 260 

(baseline tumor microenvironment composition, neoadjuvant immunotherapy response, and clinical 261 

progression). Batch extraction used OpenAI gpt-5.1 at high reasoning effort, both for Submod-262 

elTargets and for the batch-pipeline portion of CalibrationTargets; interactive CalibrationTarget 263 

extraction and all curation used Anthropic claude-opus-4-6 through Claude Code. All extraction 264 

and curation runs were performed in January and February 2026. This study used only previously 265 

published, aggregate literature data and did not involve human subjects or animal experimentation, 266 

so institutional review board or IACUC approval was not required. 267

# 3 Results 271 

3.1 SubmodelTarget Extraction and Validation 272 

We applied MAPLE to extract SubmodelTargets for a PDAC QSP model, covering tumor growth, 273 

stromal dynamics, immune recruitment, and cytokine signaling. All 37 curated SubmodelTargets 274 

used for joint inference were produced by batch extraction through the automated retry loop, 275 

followed by curation. Of these, 18 were traced with Logfire and supply the per-target retry, duration, 276 

and token metrics in Table 1; the remaining extractions predate that instrumentation and so are not 277 

in the table. The instrumented batch and the inference set therefore overlap but are not identical: 278 

a few instrumented runs targeted parameters that were consolidated or dropped during curation. 279 

The schema-validation and curation results in Section 3.2 cover all 37 targets. 280 

None of the extractions passed validation on the first attempt; all required at least one au-281 

tomated retry. Seven targets succeeded after a single retry, three each after two, three, and five 282 

retries, and one each after six and seven retries; the ECM and CCL2 secretion parameters and 283 

PSC activation required as many as 7 iterations to resolve unit ambiguities or conflicting literature. 284 

Each extraction took 7.9 minutes on average, consuming 217,735 tokens per target (3.9M total). 285 

Across the 50 retries, the validation exceptions captured in tracing (Table 2) were dominated by 286 

unit errors, followed by prior specification, then fabrication (invalid DOIs) and code errors. The 287 

LLM used 210 web searches and 22 code executions. 288

3.4 Code Generation and Inference 347 

All 37 SubmodelTargets were translated to a joint Julia inference script targeting Turing.jl 25 . The 348

# 4 Discussion 360 

Using MAPLE on the PDAC model, we extracted and curated 37 SubmodelTargets and 45 Cal-361 

ibrationTargets. Every extracted value is tied to a verbatim source snippet and a DOI-resolved 362 

citation, every observable carries unit-checked code, and 11 of the 19 unique parameters identi-363 

fied across the corpus appear in more than one target. Per-target manual extraction leaves such 364 

constraints scattered across separate documents. MAPLE binds them by shared parameter name, 365 

fitting each shared parameter against every source that constrains it in a single inference. The 366 

corpus, schemas, validators, prompts, and reference database are pinned at the commit reported 367 

in the Data Availability Statement, so the analysis can be re-run from the public repository. This 368 

matches the IQ Consortium’s recommendations for explainability and documented provenance in 369 

AI-assisted modeling 7.370 

MAPLE demonstrates that structured schemas can serve as a productive collaboration interface 371 

between LLMs and modelers for QSP model calibration. The curation analysis shows that LLM-372 

generated drafts are competent at the bibliographic, transcription, and unit-handling tasks the 373 

schema encodes, while a substantive minority of fields, concentrated in source selection, prior 374 

specification, and forward- or observable-model choice, were revised by the modeler. Rather than 375 

automating away the modeler, the framework restructures the collaboration: the schema defines 376 

what must be captured, and validators catch errors from both LLM and human sources. 377 

4.1 Validation as the Core Mechanism 378 

Across the 18 instrumented SubmodelTarget extractions, none passed validation on the first at-379 

tempt: the validators triggered 50 retries before any human review, spanning unit errors, fabricated 380 

citations, and value-snippet mismatches. The retry loop resolved most issues without human in-381 

tervention, with extractions requiring multiple retries (up to 7) typically involving ambiguous unit 382 

conventions or conflicting literature values. The same automated retry pipeline was used for batch-383 

extracted CalibrationTargets, though per-CT retry counts were not instrumented (the Logfire-384 

instrumented pipeline post-dated the CalibrationTarget extraction runs; the gap is documented in 385 

Section 3.3). 386

4.2 Schema Design Choices 402 

The separation of data extraction from model specification clarifies where human judgment is 403 

required (the inputs should be objective; the calibration layer involves modeling choices), enables 404 

reuse (if a modeling assumption changes, the inputs remain valid), and supports independent review 405 

of data versus modeling decisions. The schema and validators also work without an LLM. A modeler 406 

can fill in the fields by hand and still get the provenance trail, the automatic checks, and the route 407 

to inference code, and the validators flag transcription, unit, and reference errors no matter who 408 

entered the data. 409 

The dual-schema architecture applies the same data-first principles at two scales of calibration. 410 

SubmodelTarget operates on isolated experiments that constrain individual parameters through 411 

simplified forward models; CalibrationTarget operates on clinical and in vivo endpoints that con-412 

strain the integrated model through full-species observables. 413

4.5 Limitations and Future Work 454 

Several limitations merit discussion. First, the framework relies on the LLM’s built-in web search for 455 

source discovery, so extraction quality is bounded by the LLM’s ability to locate and access papers. 456 

Closed-access publications may be only partially available (e.g., abstract only), limiting extraction 457 

depth. Emerging agentic capabilities (LLM-controlled browsers with institutional authentication) 458 

may address this. Work in progress extends the framework to a staged extraction pipeline that 459 

retrieves full-text papers in advance and batches extraction over them systematically, which would 460 

make the external snippet-in-paper check part of the automated gate rather than an on-demand 461 

step. The framework also depends on evolving LLM capabilities: extraction is stochastic, and 462 

performance characteristics vary across providers and model versions. 463 

Second, the framework still needs substantial modeler input. For SubmodelTargets, the modeler 464 

changed the forward model type in 65% of files and adjusted prior parameters in 46%. Prompt 465 

improvements could reduce some recurring errors, but the underlying scientific decisions are likely to 466 

remain with the modeler even as LLM capabilities improve. CalibrationTargets required different 467 

curation: observable code (changed in 39% of batch-extracted files), Monte Carlo distribution 468 

derivation, and denominator audit fields demand modeling expertise that current LLMs provide as 469 

a starting point. 470 

Third, extracting structured quantitative data from figures remains challenging; the neoadjuvant 471 

immunotherapy targets required manual digitization of scatter plots. Integrating multimodal LLM 472 

vision capabilities is a natural direction for future work. Fourth, per-target extraction metrics are 473
